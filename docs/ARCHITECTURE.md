## Core Components

### Task Queue (`sugar/storage/work_queue.py`)

The work queue is an async SQLite database (via `aiosqlite`) that persists all tasks across restarts. Each work item carries a type (`bug_fix`, `feature`, `test`, `refactor`, `documentation`), a priority from 1 (urgent) to 5 (minimal), a status (`pending`, `in_progress`, `completed`, `failed`), and structured context for the executor. Tasks are retrieved in priority-then-age order using a composite index on `(priority ASC, status, created_at)`. The schema is forward-compatible through a series of async migration helpers that add columns as features land, so existing databases upgrade in place on first startup.

### Memory Store (`sugar/memory/store.py`)

The memory store is a SQLite database with optional vector search via the `sqlite-vec` extension. At startup it probes for `sqlite-vec`; if unavailable it falls back to SQLite FTS5 keyword search, then a LIKE scan. Embeddings are generated by `sugar/memory/embedder.py` (sentence-transformers when available, a deterministic fallback otherwise) and stored as packed binary floats. The store supports seven memory types (see Memory Model section below) and tracks access frequency and expiry per entry. Two scopes exist: project-local (`.sugar/memory.db`) and global (`~/.sugar/memory.db`).

### MCP Server (`sugar/mcp/`)

Sugar ships three MCP server modules: `server.py` exposes issue analysis, response generation, codebase search, and similar-issue detection; `memory_server.py` exposes the memory store (store, recall, search); and `task_server.py` exposes the work queue. Any MCP-compatible AI tool - Claude Code, GitHub Copilot, Cursor - can connect to these servers and read/write Sugar's memory or queue tasks without running the Sugar loop.

### Executor (`sugar/executor/`)

The primary executor is `AgentSDKExecutor` (Sugar 3.0+), which drives the Claude Agent SDK directly from Python rather than spawning a subprocess. It supports dynamic model routing via `sugar/orchestration/model_router.py` - simpler tasks use a lighter model tier, complex tasks are routed to a more capable one. A legacy `ClaudeWrapper` (subprocess-based) is retained for compatibility and selectable via `claude.executor: legacy` in config. The executor integrates with Ralph Wiggum (`sugar/ralph/`), an iterative self-correction loop that re-prompts Claude up to `max_iterations` times until a completion signal (`<promise>DONE</promise>`) is detected or quality gates pass.

### Quality Gates (`sugar/quality_gates/`)

The quality gates coordinator runs a pipeline of checks before any task is marked complete: preflight checks (environment sanity), test execution and validation, success criteria verification against declared acceptance criteria, truth enforcement (claims must be backed by evidence), diff validation (changes are coherent and scoped), functional verification, and a self-verification gate. Any failing gate produces a structured failure report that feeds back into the executor for a retry or escalation. Results are persisted on the work item for post-mortem review.

### Discovery (`sugar/discovery/`)

Four discovery modules populate the queue automatically when the loop runs. `GitHubWatcher` polls a configured repository for issues matching a label filter, using either the GitHub CLI (`gh`) or PyGithub. `ErrorLogMonitor` tails JSON and plain-text log files for exception patterns. `CodeQualityScanner` walks source files looking for complexity, style, and lint issues. `TestCoverageAnalyzer` identifies source modules with low or missing test coverage. Each module is independently enabled in config and adds work items with appropriate type and priority.

### Core Loop (`sugar/core/loop.py`)

`SugarLoop` is the top-level orchestrator. On each cycle it runs three phases sequentially: (1) call every enabled discovery module and enqueue new items, (2) pull the highest-priority pending item and hand it to the executor, (3) run `FeedbackProcessor` and `AdaptiveScheduler` to update priorities and record outcomes in the memory store. The cycle interval is configurable (`loop_interval`, default 300 seconds). Shutdown is cooperative - a `shutdown_event` is checked between phases and inside the sleep loop in 1-second chunks so in-flight tasks complete cleanly.

## Data Flow

How a task moves from discovery to a merged PR:

1. **Discovery** - A GitHub issue labeled "bug" appears in the repository. `GitHubWatcher` fetches it on the next poll and inserts a `bug_fix` work item with priority 2 into the queue.
2. **Queue** - The work item sits in SQLite with `status = pending`. The CLI command `sugar list` shows it.
3. **Executor pickup** - The core loop's `_execute_work` phase fetches the highest-priority pending item and transitions it to `status = in_progress`.
4. **Model routing** - `ModelRouter` inspects the task type and description length to select a model tier.
5. **Agent execution** - `AgentSDKExecutor` builds a structured prompt from the work item context, memory recalls, and acceptance criteria. It invokes the Claude Agent SDK.
6. **Ralph iteration** (if enabled) - Claude's output is checked for the completion signal. If absent, the loop re-prompts with current file state appended, up to `max_iterations`.
7. **Quality gates** - `QualityGatesCoordinator` runs the full pipeline: preflight, tests, criteria, truth, diff, functional, self-verification. All gates must pass.
8. **Git operations** - `GitOperations` commits the changes on a branch named per the pattern in config (e.g., `sugar/issue-42`).
9. **PR creation** - If `git_workflow: pull_request` is configured, a PR is opened via GitHub API linking the original issue.
10. **Outcome stored** - The task is marked `completed`, timing recorded, and an `outcome` memory entry written to the store for future similar tasks.
11. **Feedback loop** - `FeedbackProcessor` updates success rates; `AdaptiveScheduler` adjusts future priority weights.

## Memory Model

Six standard memory types plus one global type:

| Type | Purpose | Example |
|------|---------|---------|
| `decision` | Architectural and implementation choices | "We use UUIDs as primary keys, not auto-increment integers" |
| `preference` | Persistent user coding preferences | "Use `ruff` for linting, not `flake8`" |
| `file_context` | What specific files and modules do | "sugar/core/loop.py is the main orchestration entrypoint" |
| `error_pattern` | Recurring bug patterns and their fixes | "aiosqlite needs explicit `await db.commit()` after writes" |
| `research` | API docs, library findings, external knowledge | "GitHub REST API rate limit is 5000 req/hr with auth" |
| `outcome` | Task completion results and learnings | "Issue #42 fixed in 2 iterations; root cause was missing index" |
| `guideline` | Cross-project standards (global scope only) | "All public endpoints must validate input with Pydantic" |

**Search strategy:** When a query arrives, Sugar attempts vector similarity search first (cosine distance via sqlite-vec). If the extension is not loaded it falls back to FTS5 full-text search. If FTS5 returns no results it falls back to a LIKE scan. Results are ranked by similarity score multiplied by an importance weight and recency factor.

## Directory Structure

```
sugar/
  core/          Core loop and orchestration entry point
  storage/       Work queue, issue response tracking, task type registry
  memory/        Memory store, embedder, retriever, type definitions
  mcp/           MCP server modules (memory, tasks, general)
  executor/      AgentSDKExecutor, legacy ClaudeWrapper, hook runner
  ralph/         Iterative self-correction loop and completion signals
  discovery/     GitHub watcher, error monitor, code quality, test coverage
  quality_gates/ Full verification pipeline (9 modules + coordinator)
  orchestration/ Model router, workflow orchestrator
  learning/      Feedback processor, adaptive scheduler
  integrations/  GitHub API client
  agent/         SugarAgent and agent configuration
  config/        Config loading and IssueResponderConfig
  profiles/      Agent profile definitions
  triage/        Issue triage helpers
  billing/       (reserved)
  utils/         Git operations, shared helpers
  workflow/      WorkflowOrchestrator
  main.py        CLI entry point (Click)
```
